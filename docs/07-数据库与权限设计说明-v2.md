# 数据库与权限设计说明 v2（详细版）

## 1. 设计目标
- 失物、招领分表建模，避免字段语义混淆。
- 认领申请独立建模，支持“失物找招领 / 招领找失主”双向流程。
- 登录态持久化，支持每次请求校验用户会话与权限。
- 通知记录独立建模，支持站内信、短信、公众号等多通道扩展。

## 2. 核心表说明

### 2.1 `tb_user` 用户主表
- 账号域：`username`（登录名）、`phone`、`student_id`、`email`
- 身份域：`role`（`admin/user`）、`status`（`active/frozen`）
- 画像域：`college`、`major`、`grade`
- 信用域：`credit_score`、`publish_count`、`success_return_count`
- 安全域：`last_login_at`、`last_login_ip`

### 2.2 `tb_user_session` 登录会话表
- `token` 唯一，保存登录态
- 生命周期：`issued_at`、`expired_at`、`last_access_at`
- 风险审计：`login_ip`、`user_agent`
- 状态控制：`status`（`active/disabled/expired`）

### 2.3 `tb_lost_item` 失物表
- 基础信息：`title/category/sub_category/description`
- 丢失信息：`lost_time/lost_location/location_lat/location_lng`
- 联系信息：`contact_phone/contact_wechat`
- 多模态信息：`image_url/image_urls/ai_tags/manual_tags/ocr_text`
- 运营信息：`status/review_status/urgency_level`
- 生命周期：`published_at/withdrawn_at/closed_at`

### 2.4 `tb_found_item` 招领表
- 结构与失物表对齐，增加：
  - `storage_location`（保管地点）
  - `pickup_method`（认领方式）
  - `found_time/found_location`

### 2.5 `tb_claim_application` 认领申请表（关键）
- 双向关联：
  - `lost_item_id`（可空）
  - `found_item_id`（可空）
  - `application_type` 标识流程方向
- 申请信息：
  - `applicant_user_id/applicant_name/applicant_student_id/applicant_phone`
  - `evidence_text/evidence_images`
- 审核信息：
  - `status`、`reviewer_id`、`review_note`、`reviewed_at`
- 匹配依据：
  - `match_score`、`reason_summary`

### 2.6 `tb_notification_record` 消息通知记录表
- 通知对象：`user_id`
- 业务关联：`biz_type/biz_id`
- 渠道：`channel`（站内信/短信/公众号）
- 发送状态：`send_status`、`error_message`
- 阅读状态：`is_read/read_at`

### 2.7 `tb_notice` 公告表
- 内容域：`title/content/notice_type/scope/priority`
- 发布域：`status/publish_at/expire_at`
- 审计域：`created_by/updated_by/created_at/updated_at`

### 2.8 `tb_hotspot` 热点图点位表
- 点位定位：`x/y`
- 统计值：`weight/pickups`
- 分类：`point_type`（`teaching/facility/dorm`）
- 时间：`source_date`

## 3. 权限与登录态校验
- 拦截器：`AuthInterceptor`
- 放行接口：`/api/user/login`、`/api/auth/register`
- 其余 `/api/**` 必须携带 `Authorization: Bearer <token>`
- `/api/admin/**` 额外要求 `role=admin`
- 会话校验逻辑：
  1. token 存在且状态 `active`
  2. 会话未过期
  3. 用户状态为 `active`
  4. 通过后更新 `last_access_at`

## 4. 关键设计决策
- 分表优先于单表：提升语义清晰度与后续规则可扩展性。
- 会话落库优先于纯前端 token：便于强制下线和审计。
- 认领申请独立：适配复杂审核流程、证据留存与运营分析。
- 通知记录留痕：支持失败重试与读状态统计。

## 5. 现阶段限制与下一步
- 现阶段密码为明文存储（便于联调），下一步应切换 BCrypt。
- 当前权限为角色级，下一步可扩展到菜单级/资源级 RBAC。
- 当前未实现消息推送任务队列，下一步可接入异步发送与重试策略。
